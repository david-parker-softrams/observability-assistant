TOOLS DOCUMENTED
================================================================================

Tool 1: list_log_groups
  Name: list_log_groups
  Purpose: Discover available CloudWatch log groups
  Category: Inventory & Discovery
  Required Params: None (all optional)
  Optional Params: prefix, limit
  Returns: List of log groups with metadata
  Implementation: src/logai/core/tools/cloudwatch_tools.py (lines 13-125)

Tool 2: fetch_logs
  Name: fetch_logs
  Purpose: Fetch log events from specific CloudWatch log group
  Category: Search & Fetch
  Required Params: log_group, start_time
  Optional Params: end_time, filter_pattern, limit
  Returns: Sanitized log events with metadata
  Implementation: src/logai/core/tools/cloudwatch_tools.py (lines 127-310)

Tool 3: search_logs
  Name: search_logs
  Purpose: Search across multiple CloudWatch log groups
  Category: Search & Fetch
  Required Params: log_group_patterns, search_pattern, start_time
  Optional Params: end_time, limit
  Returns: Events organized by source log group
  Implementation: src/logai/core/tools/cloudwatch_tools.py (lines 312-521)

KEY FEATURES DOCUMENTED
================================================================================

1. Security: PII Sanitization
   ✓ Automatically redacts sensitive data
   ✓ Email addresses, credit cards, IPs, phone numbers, API keys, SSNs
   ✓ Configurable (enabled/disabled)
   ✓ Redaction counts returned

2. Performance: Caching
   ✓ Enabled by default
   ✓ 10-100x faster for repeat queries
   ✓ Configurable TTL (default 5 minutes)
   ✓ Transparent to agent

3. Resilience: Auto-Retry
   ✓ Detects empty results
   ✓ Expands time ranges automatically
   ✓ Suggests alternatives
   ✓ Max 3 retry attempts

4. Flexibility: Time Formats
   ✓ ISO 8601: 2024-01-15T10:30:00Z
   ✓ Relative: 1h ago, 30m ago, 2d ago, yesterday, 1w ago
   ✓ Epoch: 1705315800000

5. Filtering: Pattern Support
   ✓ Simple patterns: ERROR, "timeout"
   ✓ JSON patterns: { $.level = "ERROR" }
   ✓ Complex patterns: { ($.level = "ERROR") && ($.service = "auth") }

COMMON USAGE PATTERNS DOCUMENTED
================================================================================

Pattern 1: List → Fetch
User: "Show Lambda logs from today"
Steps:
1. list_log_groups(prefix="/aws/lambda/")
2. fetch_logs(log_group="...", start_time="1d ago")
3. Respond with analysis

Pattern 2: Search for Issue
User: "Find all 500 errors in the last hour"
Steps:
1. search_logs(patterns=["/aws/"], pattern="500", start_time="1h ago")
2. Organize by service
3. Respond with findings

Pattern 3: Cross-Service Correlation
User: "Something went wrong, investigate"
Steps:
1. search_logs(patterns=["/aws/"], pattern="ERROR", start_time="30m ago")
2. View events_by_group
3. Correlate failures

Pattern 4: Automatic Retry
User: "Any errors?"
Steps:
1. fetch_logs(...) → No results
2. Orchestrator detects empty
3. Retry with expanded time ranges
4. Eventually finds data or reports empty

================================================================================
PERFORMANCE CHARACTERISTICS DOCUMENTED
================================================================================

Typical Response Times:
- list_log_groups: 100-500ms (very fast)
- fetch_logs (1h): 200-800ms (depends on volume)
- fetch_logs (24h): 1-5s (may need filter)
- search_logs (3 groups): 500-2s (parallel)
- search_logs (10+ groups): 2-10s (scales linearly)

Optimization Tips Provided:
1. Use narrow time ranges (1h vs 24h)
2. Add filter patterns (ERROR vs no filter)
3. Specify exact log groups (vs prefixes)
4. Leverage caching (repeat queries)
5. Use limit to cap results

Cache Performance:
- First call: CloudWatch API → Cache miss → Slow
- Repeat call: Cache hit → 10-100x faster
- Different params: Cache miss → Slow
